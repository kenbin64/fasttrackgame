Rethinking Databases

Traditional databases force you to think in tables, rows, and joins. You write queries. You wait for scans. You optimize indexes. But what if data already knew where it was?

In DimensionDB, position IS the value. Every piece of data exists at a dimensional coordinate. No searching. No scanning. Just invocation.

SQL vs Dimensional

‚ùå Traditional SQL

-- Find car #75's spark plug gap
SELECT sp.gap 
FROM parking_lots pl
JOIN cars c ON c.lot_id = pl.id
JOIN engines e ON e.car_id = c.id
JOIN cylinders cy ON cy.engine_id = e.id
JOIN spark_plugs sp ON sp.cylinder_id = cy.id
WHERE pl.name = 'Main Lot'
  AND c.number = 75
  AND cy.number = 5;

-- O(n) scan + multiple joins
-- Complex query planning
-- Index optimization required

‚úÖ DimensionDB

# Invoke the coordinate directly
gap = db.invoke(
    "main-lot:car-75:cyl-5:spark-plug:gap"
)

# O(1) - constant time
# No joins (wholes contain parts)
# No query planning
# No indexes needed
# The coordinate IS the query
